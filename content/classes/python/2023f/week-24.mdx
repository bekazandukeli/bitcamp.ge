#### პროექტი 135 - Auction Platform

შექმენით Rest API DRF-ის გამოყენებით, სხვადასხვა პროდუქტების აუქციონების ბაზაზე. პლატფორმას უნდა ჰქონდეს რეგისტრაცია/დალოგინების ფუნქციონალი (შეგიძლიათ გამოიყენოთ რაიმე ბიბლიოთეკა). პლატფორმას უნდა გააჩნდეს real time ბიდინგის ფუნქციონალი WebSocket-ების გამოყენებით. როდესაც აუქციონი დასრულდება, გამარჯვებულს უნდა გაეგზავნოს email შეტყობინება შაბლონური ტექსტით მოგებული პროდუქტის შესახებ.

ტექნოლოგიები და იარაღები:

- Django Rest Framework (DRF) API-ის ასაწყობად.
- Django Channels websocket-ების იმპლემენტაციისთვის.
- კონტეინერიზაცია Docker-ისა და docker-compose გამოყენებით.
- JWT აუტენტიფიკაცია.
- Postgres მონაცემთა ბაზისთვის.
- ნებისმიერი გარე email service მეილების გასაგზავნად. (შეგიძლიათ თქვენი სერვისიც შექმნათ)
- თქვენი საბოლოო პროექტი უნდა ეშვებოდეს ერთი docker-compose ბრძანებით. ფაილში უნდა იყოს გათვალისწინებული როგორც nginx სერვისი, postgres ბაზის და თქვენი API-ს. პროექტი უნდა ეშვებოდეს gunicorn-ის მეშვეობით და არა runserver-ით.

გაითვალისწინეთ, რომ websocket-ებით უნდა გაკეთდეს მხოლოდ bidding ფუნქციონალი, რომლის აღწერა შემდეგია:

- ადმინისტრატორს (რომელიც არის superuser-ი და ჯანგოს ჩაშენებული ადმინ პანელიდან ოპერირებს) შეუძლია დაამატოს ახალი პროდუქტი და გახსნას მასზე აუქციონი.
- აუქციონს აქვს წინასწარ განსაზღვრული დრო, მინიმალური ბიჯი და ტაიმერი.
- დალოგინებულ მომხმარებლებს შეუძლიათ აუქციონებზე 'დაბიდვა' შეთავაზებული თანხით, რომელიც უფრო დიდი უნდა იყოს, ვიდრე წინა მომხმარებლის ბიდი, თუმცა, არა ნაკლები პროდუქტის მინიალურ ბიჯზე.
- როდესაც მომხმარებელი განათავსებს ბიდს, უნდა ჩაირთოს 1 წუთიანი ტაიმერი. თუ რომელიმე სხვა მომხმარებელმა ამ 1 წუთის განმავლობაში განათავსა ბიდი, მაშინ ტაიმერი თავიდან ირთვება.
- ეს ყველაფერი ხდება იქამდე, სანამ 1 წუთიანი ტაიმერი ბოლომდე არ ამოიწურება ისე, რომ ერთი მომხმარებლის განთავსებული ბიდი შენარჩუნებული იყოს.


#### პროექტი 136 - Blackjack Card Game API

შექმენით პლატფორმა, Blackjack-ის თამშის API. გთხოვთ გაეცნოთ ამ თამაშის წესებს. 
პლატფორმის მეშვეობით ჩვენ ვეთამაშებით კომპიუტერს. მთავარი ფუნქციონალი ამ პროექტისა იქნება:
- ბანქოს მენეჯმენტი (აჩეხვა, დარიგება)
- მოთამაშის და დილერის ქმედებები (hit, stand)
- ხელში არსებული კომბინაციების დათვლა
- თამაშის საბოლოო outcome-ების გადაწყვეტა (win, lose, tie)

ამ პროექტის მთავარი მიზანი არ არის შეკრათ მთლიანი ტექნოლოგიური სთექი, რასაც აქამდე ვსწავლობდით, არამედ მთავარი ფოკუსი იქნება გასაკეთებელი DRF-ის ტექნიკურ შესაძლებლობებზე.
მოგიწევთ შექმანთ დეკორატორები, კომპლექსური სერიალაიზერები და view-ები. დამატებით გასაკეთებელი იქნება რეგისტრაცია/აუტენტიფიკაციის სისტემა, თითოეული მომხმარებლისთვის წაგება/მოგების ისტორიის აღსაღრიცხად.

ფუნქციონალი:

მოიფიქრეთ პროექტის სტურქტურა და ლოგიკა.
- ბანქოს მენეჯმენტი - თამაშის დაწყებისას არსებული ბანქო უნდა 'იჩეხებოდეს'. შესამუშავებელია დარიგების ლოგიკაც. განსაზღვრეთ შესაბამისი დეკორატორები ცალკე ფაილში თქვენი view-ებისთვის და შექმნით კომპლექსური სერიალიზერების სტრუქტურა.
- თამაშის ქმედებები - განსაზღვრეთ endpoint-ები მოთამაშის ისეთი ქმედებებისთვის, როგორიცაა hit (Blackjack-ის მიხედვით, დასტიდან ახალი ბანქოს აღება) და stand (სვლის დასრულება).
- შეიმუშავეთ ხელში არსებული ბანქოების მნიშვნელობების კალკულაციის ლოგიკა.
- შეიმუშავეთ ლოგიკა თითოელი თამაშის outcome-ის გამოსარკვევად (win, lose, tie)
- მომხმარებლების აუტენტიფიკაცია jwt ტოკენების საშუალებით უნდა ხდებოდეს.
- თქვენი პროექტისთვის შეუმუშავეთ ტესტები pytest-ის გამოყენებით. 

ტექნოლოგიები:

- Django Rest Framework (DRF) API-ის ასაწყობად.
- Sqlite მონაცემთა ბაზისთვის (Postgres-იც შესაძლებელია).
- pytest და DRF-ში ჩაშენებული კლასები ტესტების გასაწერად.


#### კოლაბორაციული პროექტი 137 - Real-Time Text Editor

შექმენით google docs-ის მაგვარი კოლაბორაციული text editor-ი, სადაც რამდენიმე მომხმარებელს ერთად შეეძლებათ ტექსტის დაწერა.

ტექნოლოგიები:

- Django Rest Framework (DRF) API-ის ასაწყობად.
- Postgres მონაცემთა ბაზისთვის.
- Django Channels ვებ სოკეტებისთვის.
- Docker კონტეინერიზაციისთვის.
- Pytest ტესტებისთვის.
- Redis ქეშირებისთვის.
- JWT ტოკენები რეგისტრაცია/ავტორიზაციისთვის.

რეგისტრაცია/ავტორიზაციის შემდეგ, მომხმარებელს უნდა შეეძლოს ახალი ფაილის დამატება, სადაც შეეძლება ტექსტის აკრეფა. ამავდროულად, უნდა შეეძლოს ამ ფაილის გაზიარება სხვა მოხმარებლებისთვის, რომლებსაც სინქრონულად შეეძლებათ ტექსტის აკრეფა ფაილის ავტორთან ერთად. მოკლედ რომ ვთქვათ, google docs-ის პრიმიტიული კლონი არის ასაწყობი.
ამ პროექტისთვის შესაძლოა დაგჭირდეთ მარტივი front-ის აწყობაც, რათა გატესტოთ თქვენი API.


#### კოლაბორაციული პროექტი 138 - Project Management Tool API

ამ პროექტის ფარგლებში უნდა შექმნათ პროექტების მენეჯმენტის API, რომელიც დაეხმარება გუნდებს დაგეგმონ პროექტები, გადაანაწილონ დავალებები, ქვედავალებები, ამ დავალებების პრიორიტეტები და კომენატრები.
API-ს უნდა შეეძლოს დავალებების (Task-ების) გადანაწილება კონკრეტულ მომხმარებლებზე, მათი პროგრესის აღრიცხვა, გუნდის შორის კომუნკაციის შემუშავება, ქვეთასქების განსაზღვრა და სხვა.
აპლიკაცია გამოიყენებს Redis-ს ხშირად გამოყენებადი მონაცემების ქეშირებისთვის და Docker-ს კონტეინერიზაციისთვის.
პროექტი უნდა იყოს პოპულარული მენეჯმენტის აპლიკაცია Trello-ის გამარტივებული ანალოგი.
